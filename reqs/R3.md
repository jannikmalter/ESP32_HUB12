# R3 — ≥44 Hz content frame rate (and the architecture decision behind it)

**Requirement.** The system shall sustain a content/motion frame rate of ≥44 Hz to
the full 96×128 wall. (Panel flicker-refresh is a separate concern — see R8.)

## Rationale
44 Hz is the smooth-motion target and the legacy DMX/ArtNet full-universe ceiling.
"Content rate" (how often new frames arrive) is distinct from "panel refresh" (how
fast the driver re-scans to avoid flicker). Both must be healthy; this item covers
the former. Traces to **G2**.

## The numbers (verified — see `docs/overview.md` §7)
- Bits/frame **per chain** = 96 × 16 = **1 536**, independent of how many chains run
  in parallel (parallel data shifts one bit into all chains per clock edge).
- 1-bit refresh ceiling ≈ f_clk / 1 536 → ~6.5 kHz at 10 MHz (derate ~30%).
- Old DIP/HC modules → clean clocking realistically **~10–15 MHz**.
- 44 Hz content is far below any of these ceilings; the real constraint is the
  Ethernet/ArtNet ingest and the driver architecture, not shift bandwidth.

## Acceptance criteria
- [ ] Full wall shows ≥44 distinct frames/s from the network source, measured end to end.
- [ ] No visible tearing or frame-rate collapse under sustained streaming.

## Central architecture decision — DECIDED 2026-06-24
**Chosen: 1× ESP32-S3 parallel, 4-bit grayscale**, on a **Waveshare ESP32-S3-ETH**
board (onboard W5500 SPI Ethernet; 8 MB octal PSRAM present but not needed by the
driver — the 4-bit DMA buffer is ~24–92 KB, well within 512 KB internal SRAM). One node
drives all 8 chains via the LCD_CAM peripheral. Pin budget confirmed feasible (14 clean
GPIOs for 13 signals; GPIO33–37 reserved by the octal PSRAM regardless of use) — see
`info.md`. Implies forked parallel-output firmware. The 8× ESP32 option below is
recorded but **not taken**.

Original trade table (both paths reach the same per-chain refresh):

| Option | Effort | Trade |
|---|---|---|
| **8× ESP32** | low | one node/chain, 3 universes each; off-the-shelf-ish firmware; failure-isolated; ~€6/node; 8 IPs to manage |
| **1× ESP32-S3 parallel** | high | shared CLK/LAT/OE/A/B + 8 data lines; same refresh as one chain; single IP; **custom forked parallel-output firmware**; single point of failure; needs buffered, length-matched fan-out |

Library reality (verified): DMD32 / mrcodetastic HUB75-DMA are HUB12-capable only as
a starting point and have known HUB12 rough edges — the 1× parallel option means
forking I2S/LCD-CAM output. ESP32-S3 (LCD_CAM + PSRAM) preferred if going parallel.
