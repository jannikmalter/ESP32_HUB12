# R9 — Packed custom UDP protocol (high refresh)

**Requirement.** The system shall accept a packed custom UDP protocol carrying
4-bit/pixel data in per-chain packets, enabling content refresh beyond ArtNet's
~44 Hz/universe ceiling.

## Rationale
ArtNet (R6) is kept for interop with stock lighting gear but pads to 512 bytes/universe
(~11 Mbps for the wall) and is conventionally capped near 44 Hz. A packed protocol sends
only the bits the panel needs (6,144 B/frame) at ~half the bandwidth and ~100 Hz+,
giving headroom for an external renderer / NDI source (R10). Traces to **G2**.

## Design summary (full spec: `docs/protocol.md`)
- **Frame = 6,144 B** of pixel data (12,288 px × 4 bit).
- **Packetized per chain:** 8 packets/frame, **768 B pixel data each** (1,536 px ×
  4 bit) + small header → one un-fragmented Ethernet frame per packet (sub-1472 B).
- Avoids IP fragmentation (the W5500 hardware UDP stack does not reassemble fragments)
  and maps 1:1 onto the 8-chain DMA layout; a lost packet glitches one chain for one frame.
- ~5.3 Mbps on the wire at 100 Hz; 800 pps (vs ArtNet ~11 Mbps / 2,400 pps).

## Acceptance criteria
- [ ] Node receives all 8 per-chain packets and presents a coherent frame.
- [ ] Sustains ≥100 Hz content with no fragmentation and within W5500 throughput.
- [ ] Out-of-order / lost packet degrades gracefully (single-chain, single-frame glitch).
- [ ] Frame addressing prevents tearing (latch on full frame / sequence match).

## Open questions
- Frame assembly: latch on receipt of all 8 chain packets, or on a frame-sync marker?
- Sequence/versioning fields; how to handle partial-frame / partial-region updates.
- Transport hardening: pure UDP fire-and-forget vs. lightweight sequencing/ACK.
