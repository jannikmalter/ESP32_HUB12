# R1 — Live frame streaming over TCP/IP Ethernet

**Requirement.** The system shall stream frame content to the wall live over a
TCP/IP Ethernet link (not store-and-play).

## Rationale
The wall is for live, interactive installations; content is generated upstream in
real time. The original BX-5A4 only stores and replays sign content over a serial
bridge — unusable here. Traces to **G1**.

## Acceptance criteria
Verifiable conditions; all must hold before R1 is marked done in `reqs.md`.

- [ ] A host on the LAN can push frames to the node over Ethernet and see them
      on the wall with low, bounded latency.
- [ ] The full-wall frame (96×128 = 12 288 px) updates as a coherent frame (no tearing
      across chains).
- [ ] Link survives sustained streaming at the R3 frame rate without buffer overrun.

## Design notes
- Ethernet PHY options: LAN8720 (RMII, cheap, proven for ArtNet) or W5500 (SPI,
  better throughput on some boards). Decide alongside the board choice.
- Transport: ArtNet/UDP (see R6) is the likely primary path; a raw framebuffer push
  (TCP or UDP) may be simpler for a single source. ArtNet is connectionless UDP — the
  "TCP/IP" requirement is satisfied by the IP/Ethernet stack, not necessarily TCP.
- Frame buffering: double-buffer in the node; latch on full-frame receipt.

## Decided
- **PHY: W5500 over SPI** — it's onboard the chosen Waveshare ESP32-S3-ETH (pins
  MOSI=11/MISO=12/SCLK=13/CS=14/INT=10/RST=9). Throughput easily covers the wall.
- **Ingest: single node** (1× ESP32-S3 parallel, see R3) — one IP takes all inputs.
- **Two protocols, both into the same framebuffer:** ArtNet (R6, interop/~44 Hz) +
  packed custom UDP (R9, high-refresh, NDI-fed via R10). See `docs/protocol.md`.
